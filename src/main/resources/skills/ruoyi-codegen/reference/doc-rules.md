# Documentation Rules

本文件定义 API 文档生成时的默认规则。生成文档时，优先贴合项目现有文档风格、接口命名方式、错误码体系和示例格式；只有项目中没有明确模式时，才使用本文件的默认规则。

## Output Formats

需要文档时，默认生成：

- HTML 文档
- Markdown 文档

常见命名示例：

- `{模块名}-api-doc.html`
- `{模块名}-api-doc.md`

如果项目已有固定目录或命名规则，优先沿用项目现状。

## When to Generate Docs

以下情况优先生成文档：

- 用户明确要求生成 API 文档
- 任务属于完整模块生成
- 新增了一组完整接口
- 需要交付给前后端联调或测试使用

以下情况可不强制生成完整文档：

- 只是小范围局部改动
- 只改一个内部私有方法
- 用户明确说先不要文档

## Documentation Scope

文档至少应包含：

1. 模块说明
2. 接口列表
3. 请求方式与路径
4. 请求参数
5. 返回参数
6. 示例请求
7. 示例响应
8. 错误码或失败场景说明
9. 备注与使用说明

不要只罗列 URL，而缺少参数和返回结构说明。

## Module Overview

文档开头应说明：

- 模块名称
- 模块用途
- 适用对象或业务场景
- 接口前缀（如果有）

示例信息应简洁，不要写成大段空泛介绍。

## Endpoint Description Rules

每个接口建议说明：

- 接口名称
- 接口用途
- 请求方法（GET / POST / PUT / DELETE 等）
- 请求路径
- 权限要求（如果项目有）
- 备注说明

接口名称应尽量贴合业务语义，而不是只写“接口 1 / 接口 2”。

## Request Parameter Rules

请求参数说明应包含：

- 字段名
- 类型
- 是否必填
- 含义
- 示例值
- 备注（若有）

如果是分页接口，应明确说明：

- 页码参数
- 页大小参数
- 查询条件参数

如果是新增或更新接口，应明确区分：

- 哪些字段创建时必填
- 哪些字段更新时必填
- 哪些字段由系统生成，不应前端传入

## Response Parameter Rules

返回参数说明应至少包含：

- 字段名
- 类型
- 含义

如果项目有统一返回包装，优先按统一结构写清楚，例如：

- code
- message
- data

不要只写业务对象字段，而忽略外层响应包装。

## Example Rules

每个关键接口尽量提供：

- 示例请求参数
- 示例响应数据

示例要求：

- 字段尽量真实
- 命名贴近业务
- 不要全部写成 `"string"`、`123`
- 不要伪造明显不合理的数据结构

如果某些字段尚未最终确定，可在示例旁加备注说明待对齐。

## Error Code Rules

如果项目已有统一错误码体系：

- 优先复用现有错误码
- 说明错误码含义和触发场景
- 不要新造一套风格冲突的错误码文档

如果项目没有统一错误码体系，也应至少列出常见失败场景，例如：

- 参数非法
- 数据不存在
- 权限不足
- 状态不允许操作

## HTML Document Rules

HTML 文档应具备：

- 清晰标题层级
- 接口分节展示
- 可阅读的参数表格
- 清楚的代码示例区块
- 适合浏览器查看

不要生成只有纯文本堆叠、没有结构化排版的 HTML。

## Markdown Document Rules

Markdown 文档应具备：

- 明确标题层级
- 接口级别小节
- 参数表格
- 代码块示例
- 备注与说明区

确保 Markdown 在常见渲染器中可读，不要依赖特殊扩展语法。

## Consistency Rules

文档内容必须与当前生成代码尽量一致：

- 路径要一致
- 参数名要一致
- 返回字段要一致
- 权限标识或业务动作要一致

不要出现“代码已经改了，文档还是旧字段名”的情况。

## What to Avoid

不要这样做：

- 只列接口名，不写参数和返回值
- 示例全部用占位值敷衍
- 忽略统一响应包装
- 忽略错误码或失败场景
- 文档结构混乱，没有标题层级
- 文档内容与实际生成代码不一致

## Output Expectation

生成 API 文档时，回复中应说明：

1. 生成了哪些文档文件
2. 文档覆盖了哪些接口
3. 是否包含 HTML 与 Markdown 两种格式
4. 哪些接口说明或字段定义仍建议用户最终确认
