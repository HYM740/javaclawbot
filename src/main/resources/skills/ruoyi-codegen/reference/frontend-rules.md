# Frontend Rules

本文件定义前端代码生成时的默认规则。生成代码时，优先贴合项目现有前端目录结构、组件封装、请求封装和页面风格；只有缺少明确参照时，才使用本文件的默认规则。

## Target Stack

- Vue 3
- TypeScript
- Element Plus

如果项目中已有额外封装，例如：
- 表格组件封装
- 字典工具封装
- 权限指令
- 请求工具
- 弹窗/抽屉组件封装

优先复用，不要强行替换成另一套体系。

## Files to Generate

完整模块通常可包含：

- API 文件
- 列表页
- 表单组件
- 详情组件（有明确详情场景时）
- 类型定义（项目有单独 types 目录时）

不要为了“完整”而生成项目里并不存在的目录层级。

## Directory Strategy

默认建议位置示例：

- `src/api/{模块名}/index.ts`
- `src/views/{模块名}/{业务名}/index.vue`
- `src/views/{模块名}/{业务名}/Form.vue`

如果项目实际目录不是这样，优先沿用项目现状。

## API File Rules

### Request Layer

统一复用项目现有请求封装，例如：
- `request`
- `service`
- `http`
- 项目自定义 axios 实例

不要私自再造一套请求工具。

### API Method Naming

方法名应清晰表达业务动作，例如：

- `getAssetBaseInfoPage`
- `getAssetBaseInfo`
- `createAssetBaseInfo`
- `updateAssetBaseInfo`
- `deleteAssetBaseInfo`

如果项目已有命名习惯，例如统一使用 `list` / `detail` / `save`，优先贴合项目现状。

### Types

如果项目习惯在 API 文件内直接声明类型，就沿用；
如果项目习惯拆到独立 `types.ts`，则按项目现状拆分。

## List Page Rules

列表页通常包括：

- 搜索区域
- 操作按钮区域
- 表格区域
- 分页区域
- 新增/编辑入口
- 删除/导出等操作入口（项目已有时）

### Search Area

原则：

- 只保留真实需要的筛选项
- 不要把所有字段都堆进搜索区
- 日期范围、状态、关键字等控件类型要匹配字段语义

### Table Area

原则：

- 列定义尽量面向业务阅读
- 字段展示顺序优先贴合业务场景
- 状态值、字典值优先走项目现有字典转换方式
- 时间字段按项目现有格式化方式展示

### Actions

常见行内动作：

- 详情
- 编辑
- 删除

常见顶部动作：

- 新增
- 导出
- 批量删除（项目明确支持时）

按钮显示优先复用项目现有权限控制方式，例如：
- `v-hasPermi`
- 自定义权限组件
- 路由 meta 权限

## Form Rules

表单组件通常用于：

- 新增
- 编辑
- 查看只读（有需求时）

### Form Layout

优先使用项目已有表单布局风格：
- 单列
- 双列
- 栅格布局
- 弹窗
- 抽屉
- 独立页面

### Field Rules

- 输入控件类型要贴合字段语义
- 必填项加真实校验
- 不要所有字段都一律 `el-input`
- 字典字段优先用项目现有字典加载方式
- 日期字段使用项目统一日期组件或 Element Plus 日期组件

### Submit Logic

- 区分新增与编辑
- 表单提交前做基础校验
- 接口成功后刷新列表或关闭弹窗，行为贴合项目现状
- 错误提示使用项目现有消息提示方式

## Detail / Readonly Rules

当用户明确需要详情页或只读展示时：

- 只展示有业务价值的字段
- 优先使用项目现有详情组件或描述组件
- 不要把编辑表单硬改成详情页而不做阅读优化

## Dict and Enum Rules

如果项目已有字典系统：

- 优先复用已有字典获取方法
- 优先复用已有字典标签组件
- 不要手写一堆硬编码 `if/else` 或魔法值映射

## Permission Rules

如果项目已有权限体系：

- 复用已有权限指令或权限判断方式
- 菜单权限、按钮权限、页面权限尽量保持一致
- 不要只生成 UI，不加任何权限控制占位

## Route-related UI Rules

如果此模块需要配套路由：

- 页面标题、菜单名、缓存设置、图标等优先贴合现有路由写法
- 不要自行设计另一套路由元数据结构

## Screenshot-to-Page Rules

当用户给的是设计图、原型图、页面截图时：

1. 先识别页面属于列表页、表单页、详情页还是组合页
2. 识别顶部筛选区、操作区、表格区、弹窗区
3. 推断字段类型与交互逻辑
4. 生成 Vue 页面骨架
5. 对接口不确定部分清晰标记 TODO

注意：

- 不要凭空捏造后端真实接口
- 可以生成合理的 API 占位方法
- 可以明确写出“待与后端字段对齐”的注释

## Post-template frontend adaptation

如果前端页面是基于模板生成的，生成后应继续检查并补齐：

- 是否需要字典数据接入
- 是否需要权限指令控制
- 是否需要复杂搜索条件
- 是否需要详情页、子表组件、联动表单
- 是否需要上传、导出、状态切换等业务交互
- 是否需要根据项目现有系统组件替换默认组件写法
- 是否需要根据菜单路由规范调整页面路径和命名

模板生成的是页面基础壳子，不代表真实交互已经完整实现。

## UX and Readability

生成页面时注意：

- 结构清晰
- 交互路径自然
- 文本标签符合中文后台管理系统习惯
- 表单校验提示可理解
- 不要堆砌无意义装饰代码

## What to Avoid

不要这样做：

- 不看项目现状，强行引入全新前端架构
- 所有字段都用 `el-input`
- 所有页面都写成同一个模板壳子
- 忽略权限控制
- 忽略字典系统
- 接口命名与页面命名完全脱节
- 没有业务上下文却伪造复杂联动逻辑

## Output Expectation

生成前端代码时，回复中应说明：

1. 生成或修改了哪些前端文件
2. 页面属于哪种类型（列表 / 表单 / 详情 / 组合）
3. 复用了哪些项目已有模式
4. 哪些接口、字典或联动逻辑仍需用户确认
