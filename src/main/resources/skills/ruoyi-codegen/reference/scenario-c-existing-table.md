# Scenario C - Existing Table Code Generation

本文件用于处理“数据库表已经存在，基于现有表结构生成或补全代码”的情况。这是代码生成技能中最典型的场景之一。

## Typical Requests

常见表达包括：

- 根据 asset_base_info 表生成代码
- 按这张表补全后端和前端
- 数据库表已经有了，帮我生成 CRUD
- 根据现有表结构生成 controller/service/mapper
- 根据表补列表页和表单页
- 给这张表补菜单 SQL、路由和文档

这类请求的重点不是重新设计表，而是正确理解现有表结构并据此生成贴合项目的代码。

## Main Goal

目标是：

1. 读取真实表结构
2. 识别业务模型类型
3. 确认生成范围
4. 生成尽量贴合项目现有风格的代码
5. 避免只做机械模板替换

## First Judgment

收到请求后先判断：

1. 用户给的是单个表，还是多个表
2. 是否可能是主子表结构
3. 是否存在 `parent_id` 或类似树形字段
4. 用户要的是全量模块，还是只补某一部分
5. 是否已具备数据库连接和项目路径

不要一看到表名就默认是单表 CRUD。

## Required Inputs

基于表生成代码通常至少需要：

- 项目路径
- 模块名或业务名
- 数据库连接信息
- 目标表名
- 用户希望生成的范围

如果这些缺失，只问最小阻塞信息。

## Table Query Rule

先用数据库工具查询真实表结构，不要假设字段。

至少应获取：

- 表名
- 表注释
- 字段名
- 字段类型
- 字段注释
- 主键
- 索引
- 外键（如果工具可获取）

如果表不存在，要明确告诉用户，而不是继续假装生成。
## Template-based generation strategy

当识别为“表已存在”的生成场景时，可优先使用 `templates/` 中的 CRUD 模板生成基础代码骨架，但必须继续判断是否需要业务化改造。

处理顺序：

1. 查询真实表结构
2. 判断是单表、主子表还是树表
3. 选择对应模板生成基础骨架
4. 再根据项目指导文档和真实业务需求补充：
   - 联表查询
   - 主子表级联逻辑
   - 树结构逻辑
   - 字典翻译
   - 权限控制
   - 错误码
   - 参数校验
   - 单元测试
   - 前端交互
   - 菜单 SQL
   - 路由配置

不要把“模板已经生成完”误当成“业务已经实现完”。

## Business Model Recognition

查表后先判断属于哪类模型。

## 1. Single Table

特征：

- 只有一个主表
- 没有明显的明细表关系
- 没有父子层级依赖

处理方式：

- 生成标准单表 CRUD
- 前端一般对应列表页 + 表单页
- 根据需要补详情、导出、菜单、路由、测试、文档

## 2. Master-Detail

特征：

- 有一张主表 + 一张或多张子表
- 子表通过主键关联主表
- 业务上存在“单据 + 明细项”

处理方式：

- 主表和子表一起建模
- Service 层考虑级联保存、更新、删除
- 前端通常有主表表单 + 子表明细编辑区
- 不要误写成两个互不关联的单表 CRUD

## 3. Tree Structure

特征：

- 存在 `parent_id`、`ancestor`、`level`、`sort` 等层级字段
- 业务是分类、目录、组织等树状结构

处理方式：

- 后端考虑树查询与层级校验
- 前端考虑树表、树选择器或级联选择
- 不要按普通单表分页 CRUD 简单处理

## Field Interpretation Rules

字段解释优先依据：

1. 字段注释
2. 表注释
3. 项目现有类似模块写法
4. 常见命名语义

如果字段含义仍然不清：

- 可以保留中性命名
- 可以在说明中提示用户确认
- 不要擅自捏造业务语义

## Generation Scope Confirmation

在全量生成前，确认需要哪些内容。默认可选范围包括：

- 后端代码
- 前端代码
- 单元测试
- API 文档
- 菜单 SQL
- 路由配置

如果用户说“全部生成”，直接按全量处理即可，不要反复逐项确认。

如果用户只说“生成后端”，不要自动扩展到前端和 SQL。

## Project Adaptation Rule

基于表生成代码时，优先贴合项目现有：

- 包结构
- 命名方式
- Mapper 基类
- Service 风格
- Controller 路由风格
- 前端页面组织方式
- 请求封装方式
- 权限控制方式
- 字典处理方式

只有项目中没有明确模式时，才使用 skill 默认规则。

## Planning Policy

以下情况建议创建计划：

- 完整生成一个业务模块
- 涉及前后端多个文件
- 需要持续执行多个步骤
- 用户明确要求先出计划

以下情况通常不需要正式计划：

- 只生成后端
- 只补几个文件
- 只做一小部分增量生成

## Suggested Execution Flow

推荐执行顺序：

1. 查询表结构
2. 识别单表 / 主子表 / 树表
3. 确认生成范围
4. 生成后端代码
5. 生成前端代码
6. 补测试与文档
7. 生成菜单 SQL 与路由
8. 汇总文件并更新记忆

如果用户只要其中一部分，流程按需裁剪。

## Common Generation Outputs

常见输出包括：

### Backend
- PO / Entity
- ReqVO / RespVO
- Mapper
- Mapper XML（需要时）
- Service
- ServiceImpl
- Controller
- Convert

### Frontend
- API 文件
- 列表页
- 表单组件
- 详情组件（有需求时）

### Others
- Test
- API docs
- Menu SQL
- Route config

## Risk Control

特别注意：

- 不要因为字段名相似就误判主子表关系
- 不要忽略项目已有基类和公共封装
- 不要把树表写成普通分页表
- 不要在表结构不完整时装作已完全理解业务
- 不要直接把数据库字段原样暴露给所有前端对象而不做筛选

## When to Ask for Confirmation

以下情况建议再确认：

- 表结构存在多个合理业务解释
- 不确定是单表还是主子表
- 存在状态字段但语义不明
- 存在字典字段但取值规则未知
- 菜单权限或路由命名需要贴合项目现状
- 用户没说清楚是只要后端还是全量

## What to Avoid

不要这样做：

- 没查表就开始生成代码
- 默认所有表都是普通单表 CRUD
- 不看项目风格，直接套固定模板
- 用户只要部分生成，你却强行全量输出
- 字段注释不清时还写得像完全懂业务

## Output Expectation

处理这类请求时，回复中应明确：

1. 当前识别为“表已存在”的代码生成场景
2. 识别出的业务模型类型
3. 需要生成的范围
4. 将创建或修改的主要文件
5. 哪些业务字段、关系或规则仍需用户确认
