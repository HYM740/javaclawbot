# Scenario A - Partial Change

本文件用于处理“局部修改”类请求。局部修改指的是：用户并不需要整套 CRUD 重做，而是希望对现有模块的某一部分进行补充、修正或扩展。

## Typical Requests

常见表达包括：

- 给 xxx 模块加一个导出接口
- 把这个查询改成分页
- 补一个 update 接口
- 给列表页加一个状态筛选
- 增加一个详情页
- 调整表单校验
- 给 mapper 补一个联表查询
- 把某个字段加到前后端
- 增加菜单 SQL
- 补测试
- 改一个接口返回结构

这些请求通常不需要重走“查表 → 全量生成 → 前后端全补”的完整流程。

## Main Goal

目标是：

- 只修改必要文件
- 尽量少打扰用户
- 尽量贴合现有代码
- 不强制扩展成完整模块重构

## First Judgment

收到请求后先判断：

1. 用户要改的是后端、前端、SQL、测试还是文档
2. 改动是否依赖现有表结构
3. 改动是否会影响多个层级
4. 是否真的需要补充计划
5. 是否缺少关键上下文

如果只是一个小改动，优先直接做，不要先拉很长计划。

## Minimal Questions Policy

局部修改只问最小阻塞信息，例如：

- 改的是哪个模块
- 相关文件或类名是什么
- 关联的是哪张表或哪个接口
- 期望行为是什么

不要把完整模块生成时那套问题全部搬过来。

## When to Read Project Context

以下情况优先读取项目现有代码风格或现有模块结构：

- 用户要求“按现有风格补”
- 修改会落在已有模块里
- 需要保持接口、命名、分页、权限、字典风格一致

如果现有上下文不足，不要凭空造风格。

## Typical Partial Change Types

## 1. Backend Partial Change

例如：

- 增加 controller 接口
- 增加 service 方法
- 增加 mapper 查询
- 增加 VO 字段
- 补 converter 映射
- 新增导出功能

处理原则：

- 从现有层级向下或向上最小扩散
- 能复用现有类就不要新建一套类
- 不因为补一个方法而重生成全部文件

## 2. Frontend Partial Change

例如：

- 列表页增加搜索条件
- 表格增加一列
- 表单增加字段
- 增加详情页弹窗
- 补按钮权限控制
- 增加导出按钮

处理原则：

- 尽量在现有页面上增量修改
- 复用现有组件、hooks、API 文件
- 不强行重写整个页面

## 3. SQL / Route / Menu Partial Change

例如：

- 补菜单 SQL
- 加一条路由
- 增加按钮权限
- 调整排序

处理原则：

- 先对齐项目现有表结构或路由格式
- 只补必要项
- 不重做整个菜单体系

## 4. Test / Doc Partial Change

例如：

- 给新加的 service 方法补测试
- 给新增接口补文档
- 更新已有文档中的字段说明

处理原则：

- 只围绕本次变更补内容
- 不为了“完整性”强制把全模块测试和文档都重写一遍

## Planning Policy

局部修改通常不需要正式计划。

只有在这些情况下才考虑建计划：

- 修改跨越多个层级，影响面较大
- 用户明确说“按步骤来”
- 改动会分阶段进行
- 需要持续跟踪未完成事项

否则直接执行更好。

## Output Strategy

局部修改的回复应尽量简洁，建议结构：

1. 当前识别的修改类型
2. 将修改哪些文件
3. 关键实现点
4. 仍需用户确认的信息（如有）

不要先输出大段项目管理内容。

## Risk Control

局部修改时特别注意：

- 不要破坏现有接口契约
- 不要随意重命名已有类和方法
- 不要把小改动扩展成大重构
- 不要引入与项目现状不一致的新框架写法

## When to Ask for Confirmation

以下情况建议先确认再改：

- 会改动已有接口入参或返回结构
- 会影响数据库字段语义
- 会影响前端已有交互
- 会改动权限标识或菜单层级
- 用户描述过于模糊，存在多个合理方案

## What to Avoid

不要这样做：

- 把“补一个方法”变成“重做整个模块”
- 只因为用户提到了表名，就强行走完整 codegen 流程
- 没看项目现有写法就直接套模板
- 问一大串完整模块生成问题
- 对局部改动输出过长计划和状态管理内容

## Output Expectation

处理局部修改时，回复中应明确：

1. 识别到这是局部修改，不是完整模块生成
2. 会增量修改哪些文件
3. 为什么这样改最贴合现有项目
4. 哪些地方仍需用户确认
