# Menu SQL Rules

本文件定义菜单 SQL 生成时的默认规则。生成菜单、页面、按钮权限相关 SQL 时，必须优先贴合目标项目真实菜单表结构、字段命名、ID 生成方式和权限标识规范；不要直接生搬硬套模板。

## Core Principle

先查真实表结构，再生成 SQL。

不要因为模板里叫 `system_menu`，就默认所有项目都一定是这个表、这些字段、这种 type 含义。

## Required Pre-check

生成菜单 SQL 前，至少先确认以下信息：

1. 菜单表名
2. 主键字段名
3. 菜单名称字段
4. 父级菜单字段
5. 路径字段
6. 组件字段
7. 权限标识字段
8. 菜单类型字段
9. 排序字段
10. 状态字段
11. 创建时间 / 更新时间字段（如果项目要求）

如果这些信息不清楚，就不要直接产出“可执行 SQL”。

## Table Discovery Rule

优先通过数据库工具查询真实表结构，例如：

```bash
node scripts/mysql-tool.js --host {host} --port {port} --user {user} --password "{password}" --database {database} --action query --table system_menu
```

如果 `system_menu` 不存在，应继续根据项目实际情况查找等价菜单表，而不是假装已经查到。

## Field Comparison Rule

把模板所需字段与真实表结构逐项对比，至少确认：

- 字段名是否一致
- 字段类型是否兼容
- 是否有非空字段必须赋值
- 默认值是否可省略
- 是否存在逻辑删除字段
- 是否存在租户字段 / 系统字段 / 组织字段等额外约束

如果真实表结构与模板不同，应按真实表结构改写 SQL。

## Menu Type Rule

只有在确认真实项目定义后，才能使用菜单类型值。

常见但不可默认套用的约定：

- 目录：`type = 1`
- 菜单页：`type = 2`
- 按钮：`type = 3`

如果项目实际不是这样，必须以真实项目定义为准。

## Parent-Child Rule

生成菜单层级时，先确认：

- 顶级目录的 parent_id 是多少
- 二级页面挂在哪个目录下
- 按钮权限是否作为页面子节点存储
- 是否支持超过三级层级

不要默认所有项目都一定是“目录 → 页面 → 按钮”三层结构。

## ID Strategy Rule

生成 SQL 前先确认 ID 生成策略：

### 情况 1：数据库自增

如果主键是自增：

- 插入时通常不手写主键
- 但如果项目要求固定菜单 ID，则按项目要求处理

### 情况 2：手工指定 ID

如果项目手工维护菜单 ID：

- 先查询当前最大 ID
- 再从最大值 + 1 开始分配
- 确保同一批生成的菜单 ID 连续且不冲突

### 情况 3：雪花 / UUID / 序列

如果项目有专门主键策略：

- 按项目策略生成
- 不要擅自改成 `MAX(id)+1`

## Permission Rule

权限标识必须贴合项目现有规范。

常见形式示例：

- `{模块}:{业务}:{操作}`
- `{系统}:{模块}:{操作}`
- `app:{模块}:{业务}:{操作}`

只有在项目现有权限规范不明确时，才使用 skill 默认格式；一旦项目已有既定格式，必须优先沿用。

## Route and Component Rule

如果菜单表中存储页面路由、组件路径、重定向、图标、缓存等信息，生成前应确认：

- 路由路径格式
- 组件路径格式
- 是否需要 `redirect`
- 是否需要 `alwaysShow`
- 是否有 `keepAlive`
- 是否有 icon/meta/title 等字段

不要只插入菜单名和权限，导致菜单虽然存在但页面无法正确加载。

## Minimal Insert Scope

默认至少区分三类记录（若项目支持）：

1. 目录
2. 页面
3. 按钮权限

但不是所有项目都必须三类全生成。

例如：

- 有些项目按钮权限不落库
- 有些项目目录和页面合并
- 有些项目纯前端路由，不需要菜单 SQL

先看项目现状，再决定范围。

## Ordering Rule

如果项目菜单支持排序：

- 生成时给出合理 sort 值
- 同一父级下排序连续
- 不要全部写成同一个值

如果项目已有菜单排序习惯，应尽量贴合现有区间和规律。

## Status and Visibility Rule

如果菜单表有状态、显示/隐藏、缓存、外链、权限开关等字段：

- 按真实表结构赋值
- 缺省值需符合项目惯例
- 不要随意写死“启用/显示/缓存”之类值

## Tenant / System-specific Rule

如果项目是多租户、多系统、分组织架构的后台：

- 注意 tenant_id / system_id / org_id / app_id 等字段
- 菜单可能并不是全局共享
- 必须确认插入范围与归属

遇到这些字段时，不能忽略不管。

## SQL Style Rule

生成 SQL 时优先贴合项目现有风格，例如：

- 单条 INSERT
- 批量 INSERT
- 先查父菜单再插子菜单
- 使用变量保存新 ID
- 使用事务包裹

如果项目没有明确风格，生成的 SQL 应做到：

- 可读
- 可执行
- 字段齐全
- 注释清晰（必要时）

## Recommended Output Structure

生成菜单 SQL 时，回复中建议说明：

1. 识别到的菜单表名
2. 菜单层级结构
3. 将生成哪些记录（目录 / 页面 / 按钮）
4. 权限标识规则
5. SQL 中哪些值是依据真实表结构决定的
6. 哪些值仍需用户确认

## What to Avoid

不要这样做：

- 没查表结构就直接写 SQL
- 默认所有项目都叫 `system_menu`
- 默认 `type=1/2/3` 一定正确
- 默认主键一定可用 `MAX(id)+1`
- 忽略路由、组件、缓存等关键字段
- 忽略租户、系统、组织归属字段
- 生成出“能插进去但前端不能用”的菜单数据

## Output Expectation

菜单 SQL 生成完成后，回复中应明确：

1. 使用的是哪个菜单表
2. 生成了哪些菜单层级记录
3. 采用了哪种 ID 策略
4. 权限标识如何命名
5. 哪些字段来源于真实表结构
6. 哪些字段仍建议用户最终确认
