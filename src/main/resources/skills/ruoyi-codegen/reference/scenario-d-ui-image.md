# Scenario D - UI Image to Vue Page

本文件用于处理“用户提供页面截图、原型图、设计稿，希望生成 Vue 页面代码”的情况。此类任务的重点不是数据库优先，而是先把界面结构、字段、交互和组件关系分析清楚，再生成贴合项目风格的前端代码；若用户还需要后端，则在后续补接口设计或代码骨架。

## Typical Requests

常见表达包括：

- 根据这张设计图生成 Vue 页面
- 按截图还原后台管理列表页
- 看这张原型图帮我写前端页面
- 按这个 UI 做一个 Element Plus 页面
- 根据图片生成列表页和表单弹窗
- 根据这个界面补 API 对接骨架

这类请求的核心是“以界面为起点”，而不是“以数据库表为起点”。

## Main Goal

目标是：

1. 分析页面用途和结构
2. 识别字段、组件和交互
3. 生成 Vue 3 + Element Plus 页面骨架或可用页面
4. 对未知接口保留清晰的对接占位
5. 尽量贴合项目已有页面风格和组件封装

## First Judgment

收到截图或设计图后先判断：

1. 这是列表页、表单页、详情页，还是组合页
2. 是否存在搜索区
3. 是否存在表格区
4. 是否存在弹窗、抽屉或子表区域
5. 是否能明显识别字段标签和按钮动作
6. 用户要的是静态页面、交互骨架，还是带 API 对接的页面

不要一上来就假设完整后端接口已经存在。

## Image Analysis Strategy

先从整体再到局部：

### 1. 页面级判断
识别页面属于：
- 数据列表管理页
- 新增/编辑表单页
- 详情展示页
- 主从组合页
- 多标签页或分步表单页

### 2. 区域拆分
优先拆出这些区域：

- 顶部筛选区
- 操作按钮区
- 表格区
- 分页区
- 弹窗 / 抽屉区
- 明细子表区
- 详情展示区

### 3. 字段识别
识别：

- 字段标签
- 输入控件类型
- 是否必填
- 是否字典选项
- 是否时间范围
- 是否状态开关
- 是否上传、富文本、树选择等特殊控件

### 4. 动作识别
识别：

- 查询
- 重置
- 新增
- 编辑
- 删除
- 详情
- 导出
- 提交
- 取消
- 审核 / 发布 / 启停 等业务动作

## Minimal Questions Policy

如果图片已经足够清楚，只问最少阻塞信息，例如：

- 这是要做静态还原，还是带 API 对接
- 页面对应的模块名是什么
- 是否需要同时生成后端接口骨架
- 是否要尽量贴合现有项目页面结构

不要在图片任务一开始就追问数据库、表结构、完整模块计划，除非后续确实需要。

## Frontend Generation Strategy

默认优先生成：

- Vue 页面文件
- API 占位文件或接口调用骨架
- 表单组件 / 明细组件（如果页面结构需要）

生成时优先贴合项目现有：

- 页面目录
- 组件写法
- hooks 风格
- 表格封装
- 请求封装
- 权限控制
- 字典调用方式

如果项目上下文不足，再使用通用 Vue 3 + Element Plus 写法。

## Common Page Patterns

## 1. List Page

典型特征：

- 搜索表单
- 操作按钮
- 表格
- 分页

应生成：

- 查询表单模型
- 表格列配置
- 查询 / 重置逻辑
- 新增 / 编辑 / 删除入口
- 分页状态与数据加载逻辑

## 2. Form Dialog / Drawer

典型特征：

- 多字段输入
- 提交 / 取消按钮
- 可能区分新增与编辑

应生成：

- 表单模型
- 校验规则
- 打开/关闭逻辑
- 提交逻辑
- 回填逻辑

## 3. Detail Page

典型特征：

- 只读展示
- 结构化字段区域
- 附件、图片、状态展示

应生成：

- 详情数据加载逻辑
- 展示布局
- 字段格式化显示
- 相关子信息区域（如有）

## 4. Master-Detail Composite Page

典型特征：

- 主表单 + 子表列表 / 明细录入区
- 一个主对象配多个子项

应生成：

- 主表单模型
- 子表数据结构
- 新增/删除子项逻辑
- 整体提交数据结构

## Unknown Backend Handling

当后端接口未知时：

- 可以生成合理的 API 方法占位
- 可以用清晰注释说明待对接接口
- 可以基于页面行为推断需要的接口类型

例如常见接口骨架：

- 分页查询
- 详情查询
- 新增
- 更新
- 删除
- 字典获取
- 上传接口

但不要假装这些接口一定已经存在，也不要伪造数据库真实字段。

## Screenshot-to-Code Detail Rules

### Field Type Guessing

根据标签和视觉控件推断字段类型时，要尽量保守：

- 文本框 → 普通输入
- 下拉框 → 字典/枚举
- 日期控件 → 日期 / 日期范围
- 开关 → 状态布尔值
- 上传区域 → 文件 / 图片
- 表格列中的标签色块 → 状态字典展示

如果不确定，可保留中性字段名或加 TODO。

### Naming Rule

命名优先级：

1. 用户已提供的业务名称
2. 页面标题 / 区域标题
3. 组件语义
4. 通用后台命名

不要从图片中凭空推断出过于具体但不可靠的业务术语。

## Style Adaptation Rule

如果项目已有后台页面模板或统一组件体系：

- 优先复用
- 不要直接换成另一套完全不同的页面组织方式
- 不要仅因为截图看起来简单，就忽视现有项目封装

## When to Ask for Confirmation

以下情况建议补问一次关键确认：

- 页面用途不明确
- 截图文字过少、字段难识别
- 不确定是静态还原还是可交互页面
- 用户同时要求后端，但未提供任何业务上下文
- 页面明显是复杂主子表或审批流界面

## Planning Policy

图片生成页面通常不一定需要正式计划。

以下情况可直接生成：

- 单页列表
- 单个表单弹窗
- 简单详情页
- 小型后台管理页面

以下情况可考虑建计划：

- 多页面成套生成
- 页面 + API + 后端一起生成
- 含主子表、审批流、复杂交互
- 用户明确要求分步骤推进

## Output Strategy

处理图片生成页面时，回复建议结构：

1. 当前识别的页面类型
2. 拆分出的主要区域
3. 将生成哪些文件
4. 哪些交互可以直接落地
5. 哪些接口或字段仍待确认

## What to Avoid

不要这样做：

- 没分析页面结构就直接套模板
- 把 UI 图任务误当成数据库表生成任务
- 接口未知时却假装后端已经存在
- 所有字段都简单粗暴用 `el-input`
- 不看项目现有页面风格，直接写一套陌生结构
- 为了显得完整而伪造复杂业务流程

## Output Expectation

处理这类请求时，回复中应明确：

1. 当前识别的是哪种页面类型
2. 页面包含哪些区域与交互
3. 将生成哪些前端文件
4. 是否包含 API 占位或对接骨架
5. 哪些字段、动作或接口仍需用户确认
