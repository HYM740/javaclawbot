# Scenario B - Full Feature but No Existing Table

本文件用于处理“用户描述了完整业务功能，但数据库表尚未存在”的情况。此时不能直接进入基于表结构的代码生成，而应先完成业务建模与表结构设计，再进入后续代码生成。

## Typical Requests

常见表达包括：

- 我要做一个 xxx 管理模块，但库里还没有表
- 帮我设计这个功能的表结构并生成前后端
- 先给我出 DDL，再生成代码
- 这个业务要新增一个模块，先设计库表
- 按这个需求设计表，然后补全 CRUD

这类请求属于“先建模，再生成”。

## Main Goal

目标是：

1. 先把业务对象、字段和关系梳理清楚
2. 生成合理的表结构 / DDL
3. 用户确认后，再继续生成代码
4. 避免在表未定的情况下直接伪造后端和前端细节

## First Judgment

收到请求后先判断：

1. 这是单表、主子表还是树形结构
2. 用户是否已经明确了核心字段
3. 是否已有现成表可以复用或扩展，而不是新建
4. 是否需要先问少量关键业务问题
5. 用户是要“先出表结构”，还是“表结构+代码一起规划”

## Minimal Questions Policy

只问阻塞设计的最少信息，例如：

- 这个模块的核心业务对象是什么
- 需要记录哪些关键信息
- 是否有主从关系
- 是否有状态、流程、审批、字典、附件等扩展需求
- 是否需要树形层级
- 是否需要软删除、租户、组织归属等系统字段

不要在一开始问过多实现细节，更不要直接问完整项目所有问题。

## Modeling Priority

设计表结构时遵循以下优先级：

1. 先理解业务对象与关系
2. 再确定字段
3. 再确定索引、唯一约束、状态字段
4. 最后再考虑代码生成映射

不要反过来为了生成代码方便，强行把业务结构设计得很奇怪。

## Common Structure Types

## 1. Single Table

适用于：

- 一个主业务对象即可表达需求
- 没有明显子项集合
- 不需要树级管理

示例：

- 资产分类配置
- 仓库信息管理
- 供应商基础资料

处理重点：

- 核心字段
- 状态字段
- 排序字段
- 备注字段
- 审计字段

## 2. Master-Detail

适用于：

- 一个主单据 + 多个明细项
- 一对多结构明显

示例：

- 处置单 + 处置明细
- 入库单 + 入库明细
- 盘点单 + 盘点明细

处理重点：

- 主表负责单据级信息
- 子表负责明细级信息
- 明确外键关系或逻辑关联字段
- 明确是否需要级联删除 / 级联查询

## 3. Tree Structure

适用于：

- 存在父子层级
- 需要递归展示
- 需要目录、分类或组织树

示例：

- 资产分类树
- 组织架构
- 区域层级

处理重点：

- `parent_id`
- 层级标识
- 排序字段
- 是否需要路径字段或祖级链路字段

## DDL Design Rules

设计 DDL 时注意：

- 字段名清晰、稳定、可维护
- 类型尽量贴合业务语义
- 长度设置合理
- 时间、状态、备注等常见字段按项目惯例补齐
- 必填字段要明确
- 唯一性和索引只加真正有业务价值的部分

不要为了“看起来完整”加太多无意义字段。

## Project-specific Fields

如果项目通常包含这些系统字段，应考虑是否需要加入：

- `id`
- `create_time`
- `update_time`
- `creator`
- `updater`
- `deleted`
- `tenant_id`
- `dept_id`
- `remark`
- `status`
- `sort`

但不要无脑全部加上。必须看业务是否需要、项目是否已有统一约定。

## DDL Confirmation Policy

在继续生成代码前，先给用户确认以下内容：

1. 表名
2. 字段清单
3. 主键策略
4. 主从/树结构关系
5. 索引或唯一约束
6. 是否沿用项目通用审计字段

如果这些还没确认，不要假装已经可以稳定生成全部代码。

## Output Strategy Before Codegen

当表还没落定时，回复建议结构：

1. 当前识别为“表未存在”的完整功能需求
2. 推断的业务模型类型
3. 建议的表结构或 DDL
4. 需要用户确认的关键点
5. 确认后将继续生成哪些代码

## Transition to Codegen

只有在 DDL 已明确后，再进入代码生成阶段。

后续可生成：

- 后端代码
- 前端代码
- 测试
- API 文档
- 菜单 SQL
- 路由配置

如果用户只要 DDL，则不要主动扩展到全部代码生成。

## Risk Control

特别注意：

- 不要凭空捏造复杂业务流程字段
- 不要在需求不明时设计过深的表关系
- 不要把简单模块强行设计成主子表
- 不要在树结构需求不明确时随意加 `parent_id`
- 不要为了代码模板方便而牺牲表设计合理性

## When to Ask for Confirmation

以下情况建议明确确认：

- 是否需要主子表
- 某些关键字段是字典还是自由文本
- 是否需要审批状态 / 发布状态 / 启停状态
- 是否需要附件、图片、富文本
- 是否需要多租户 / 部门 / 组织归属

## What to Avoid

不要这样做：

- 用户说“没有表”，你却假装已经查表成功
- 直接生成完整 CRUD，但底层表结构还没定
- 表结构设计明显只为凑模板
- 问一大串无关实现问题
- 需求稍微复杂就强行过度建模

## Output Expectation

处理这类请求时，回复中应明确：

1. 当前是“先设计表”的场景
2. 推断的业务模型是单表、主子表还是树表
3. 建议的 DDL 或字段方案
4. 哪些点必须用户确认
5. 确认后会继续生成哪些内容
